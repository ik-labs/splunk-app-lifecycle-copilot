# Splunk App Lifecycle Copilot — Scope Doc (Hackathon MVP)

**Hackathon:** Splunk Agentic Ops Hackathon (Cisco/Splunk × Devpost)
**Track:** Platform & Developer Experience
**Bonus targeted:** Best Use of Splunk MCP Server
**Submission deadline:** Jun 15, 2026, 9:00 AM PDT

---

## 1. One-line pitch

An agent that carries a Splunk app from raw logs to a CIM-clean, AppInspect-green package by running self-healing loops — diagnosing, patching, and re-validating each artifact — while leaving an auditable trail of every fix. Splunk's own AI *explains* failures; this *resolves, validates, and remembers* them.

## 2. What we are building (MVP scope)

Two fully working loops on a shared spine. Three further lifecycle stages are present in the architecture but explicitly marked not-implemented.

| Stage | Status | Needs live Splunk? |
|---|---|---|
| 1. Onboarding loop (raw log → CIM-clean extraction) | **BUILD** | Yes |
| 2. AppInspect loop (app → compliance-green) | **BUILD** | No (static analysis) |
| 3. Scaffold + test data | Architecture only | — |
| 4. Cost-aware SPL lint | Architecture only | — |
| 5. Dashboard migration | Architecture only | — |

The shared spine = orchestrator agent + self-heal engine + deterministic patcher registry + provenance ledger + MCP/SDK layer. The self-heal engine is one piece of code reused by both loops; that reuse is the platform thesis.

## 3. Build sequencing (de-risked order)

1. **AppInspect loop first.** Zero infrastructure — `pip install splunk-appinspect` and go. Guaranteed to work end-to-end. This is the safety-net deliverable.
2. **Shared self-heal engine** extracted from loop 1 (diagnose → patch → re-validate → retry, with max-iteration cap).
3. **Onboarding loop second**, reusing the self-heal engine pointed at extraction-regex artifacts instead of AppInspect failures.
4. **Provenance ledger** threaded through both.
5. **Polish, demo script, backup video.**

Rationale: if time runs out, a working AppInspect loop + the architecture is still a complete, submittable, prize-eligible project.

## 4. Technical architecture

### 4.1 Components

- **Orchestrator agent** — Python, built on the Splunk SDK `splunklib.ai` agent primitives (`AgentState`, subagents, tool calls). Plans the run, routes to the right loop, invokes the self-heal engine.
- **Self-heal engine** — generic loop: `diagnose(failure) → select_patch() → apply_patch() → re-validate() → repeat until clean or max_iters`. Artifact-agnostic, but intentionally **not** a free-form code editor. Patch application is done by deterministic patch functions from a constrained registry. The LLM produces diagnosis, rationale, and operator-facing explanation text only; it does not write raw file edits. This keeps the demo repeatable and the audit trail defensible.
- **MCP/SDK layer** — MCP-app-token-authenticated client to the Splunk MCP server. Baseline tools: `splunk_run_query` for validation searches and `splunk_get_knowledge_objects` for existing knowledge-object discovery. Optional Splunk AI Assistant tools: `saia_generate_spl`, `saia_explain_spl`, and `saia_optimize_spl`. Direct `splunklib.client` is still used for ingestion/search setup where simpler than MCP.
- **Provenance ledger** — append-only JSON log; one entry per patch: `{stage, iteration, failure, diagnosis, patch, rationale, validation_result, timestamp}`.
- **Splunk instance** — `splunk/splunk:latest` in Docker, trial or Free license.

### 4.2 Auth note

Use the **encrypted token generated by the Splunk MCP Server app**. It is not a plain Splunk REST bearer token, and encrypted MCP tokens cannot be reused for direct Splunk API calls. OAuth for MCP exists, but it is Controlled Access / closed preview and is not a build dependency for this hackathon. Store the MCP endpoint and encrypted token in `.env`, never commit them.

`splunk_run_query` is intentionally bounded by the MCP server for safe, quick searches: about a 1-minute execution limit and about 1000 events returned. The onboarding fixture has 150 lines, so it is safely inside those limits.

### 4.3 Per-loop call flows

**Onboarding loop (Stage 1):**
1. Ingest raw log sample to Splunk via HEC (`splunklib` / HEC token).
2. Draft candidate extraction logic as inline SPL (`rex`, `eval`, `fieldformat`, aliases-as-eval) plus a candidate CIM data model mapping. Use `saia_generate_spl` only when Splunk AI Assistant is installed and the tool is available; otherwise use the local LLM/rule templates and still validate through MCP.
3. Run `splunk_run_query` through MCP against the ingested sample, validating the candidate extraction against real indexed events without writing `props.conf` / `transforms.conf` or reloading Splunk per iteration.
4. **Diagnose:** compare extracted fields vs. expected CIM fields. Count coverage (e.g. "4 of 9 fields extracted, `src` not mapped"). Use `splunk_get_knowledge_objects` where helpful to inspect existing field extractions, aliases, calculated fields, and data models.
5. **Patch:** deterministic onboarding patch functions adjust the inline `rex` / `eval` / alias candidate from a constrained set of known transformations. The LLM writes the diagnosis and rationale, not the patch bytes.
6. Re-run `splunk_run_query`. Repeat until CIM-clean. Flag any PII fields detected.
7. After convergence only, emit final `props.conf` / `transforms.conf` + provenance entries.

**AppInspect loop (Stage 2):**
1. Take a deliberately non-compliant app directory.
2. Run `splunk-appinspect inspect <dir> --mode test --data-format json --output-file result.json`.
3. **Diagnose:** parse `result.json`; extract each failure (check name, file, line, message).
4. **Patch:** deterministic AppInspect patch functions map known failure types to known repairs (remove forbidden `local/` config, remove `user-seed.conf`, disable/remove forbidden forwarding in `outputs.conf`, etc.). The LLM produces the human-readable diagnosis and rationale only; it does not edit files directly.
5. Re-run AppInspect. Repeat until zero failures or max_iters.
6. Emit clean app + provenance entries.

## 5. Seed fixtures (the demo's raw material)

Both loops need *deliberately broken* inputs so there is something to heal.

### 5.1 Onboarding fixture — messy log sample

A custom-format log where field structure is non-obvious. **Recommended (on-brand, novel): a UPI / payment-gateway transaction log or GST e-invoice log.** Example shape:

```
2026-06-01T14:22:08+05:30 txn_id=UPI2026060114220812345 vpa=merchant@okhdfcbank amt=1499.00 status=SUCCESS rrn=412345678901 payer_vpa=customer@ybl gstin=24AAACC1206D1ZM
```

Designed so a naive extraction misses several fields and the CIM mapping (e.g. amount → `amount`, status → `action`/`status`) needs iteration. Include at least one field that should be flagged PII (e.g. `payer_vpa`).

Provide ~50–200 sample lines (well under the 500MB/day Free tier).

### 5.2 AppInspect fixture — non-compliant app

A minimal Splunk app seeded with three deterministic AppInspect failures:

1. **Forbidden `local/` directory** — config under `local/` triggers `check_that_local_does_not_exist`.
2. **Forbidden `user-seed.conf`** — default login/password seeding triggers `check_user_seed_conf_deny_list`.
3. **Forbidden forwarding config** — enabled `default/outputs.conf` triggers `check_if_outputs_conf_exists`.

Each has a known, scriptable fix, so the loop reliably reaches green on camera.

## 9. Risks & mitigations

- **Onboarding loop is the harder half** (regex gen + live-search validation has more moving parts) → build AppInspect loop first as the guaranteed deliverable.
- **Docker Splunk first-boot is slow (~1–2 min to "healthy") and memory-hungry on Mac** → pre-warm before any live demo; resource-limit the container.
- **Live demo gremlins** → record a clean backup video; rules allow judges to score on video alone.
- **Over-promising the 5-stage platform** → show two loops working for real; present stages 3–5 as labeled architecture. Judges reward honest scoping.
- **Direct overlap with Splunk's AI Intelligence Reports for AppInspect (early 2026)** → name their feature explicitly in README/demo; position this as the layer above ("Splunk explains; we resolve, validate, remember"). The closed loop + provenance + onboarding integration is the defensible delta.

## 10. 3-minute demo script (minute-by-minute)

**0:00–0:25 — The problem.**
Cold open on the pain: a developer's week disappears into onboarding a new log source by hand, then days fighting AppInspect's ping-pong, and the hard-won knowledge dies in one engineer's head. State the thesis: Splunk's AI explains failures; nothing closes the loop or remembers why.

**0:25–1:15 — Onboarding loop (live or recorded).**
Drop in the messy UPI/GST log. Agent ingests it, drafts inline `rex` / `eval` extraction candidates, runs `splunk_run_query`, and reports "4 of 9 fields extracted, `src` not CIM-mapped." Show the self-heal loop fixing the candidate extraction and re-running until CIM-clean — without Splunk reload churn — and flagging `payer_vpa` as PII. Land the point: suggestion vs. *verified against real events*.

**1:15–2:15 — AppInspect loop (the hero shot).**
Point the agent at the deliberately broken app. Show the three real failures (`local/` config, `user-seed.conf`, forbidden `outputs.conf`). Watch the agent diagnose → deterministic patch → re-run AppInspect → turn red to green autonomously. This is the most visceral "it actually fixed it" moment — give it the most screen time.

**2:15–2:45 — The platform thesis + provenance.**
Show the same self-heal engine drove both loops. Open the provenance ledger: every fix with its rationale and timestamp. Make the institutional-memory point — the next developer isn't lost. Flash the architecture diagram showing stages 3–5 as the roadmap.

**2:45–3:00 — Close.**
Restate value: a week of expert-gated toil compressed to an afternoon, with an audit trail. Name the MCP server's role (the agent orchestrating real actions across the lifecycle) to anchor the bonus-prize case. End on the repo URL.

**Throughout:** every time the agent calls the MCP server, make it visible on screen — the bonus judges are looking for the MCP server orchestrating meaningful actions, so show the calls, don't hide them.
