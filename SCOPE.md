## 1. One-line pitch

An agent that carries a Splunk app from raw logs to a CIM-clean, AppInspect-green package by running self-healing loops ‚Äî diagnosing, patching, and re-validating each artifact ‚Äî while leaving an auditable trail of every fix. Splunk's own AI *explains* failures; this *resolves, validates, and remembers* them.

## 2. What we are building (MVP scope)

Three fully working loops on a shared spine. Two further lifecycle stages are present in the architecture but explicitly marked not-implemented.

| Stage | Status | Needs live Splunk? |
|---|---|---|
| 1. Onboarding loop (raw log ‚Üí CIM-clean extraction) | **BUILD** | Yes |
| 2. AppInspect loop (app ‚Üí compliance-green) | **BUILD** | No (static analysis) |
| 3. Scaffold + test data | Architecture only | ‚Äî |
| 4. Cost-aware SPL lint (costly search ‚Üí clean) | **BUILD** | No (static analysis) |
| 5. Dashboard migration | Architecture only | ‚Äî |

The shared spine = orchestrator agent + self-heal engine + deterministic patcher registry + provenance ledger + MCP/SDK layer. The self-heal engine is one piece of code reused by both loops; that reuse is the platform thesis.

## 3. Build sequencing (de-risked order)

1. **AppInspect loop first.** Zero infrastructure ‚Äî `pip install splunk-appinspect` and go. Guaranteed to work end-to-end. This is the safety-net deliverable.
2. **Shared self-heal engine** extracted from loop 1 (diagnose ‚Üí patch ‚Üí re-validate ‚Üí retry, with max-iteration cap).
3. **Onboarding loop second**, reusing the self-heal engine pointed at extraction-regex artifacts instead of AppInspect failures.
4. **Provenance ledger** threaded through both.
5. **Polish, demo script, backup video.**

Rationale: if time runs out, a working AppInspect loop + the architecture is still a complete, submittable, prize-eligible project.

## 4. Technical architecture

### 4.1 Components

- **Orchestrator agent** ‚Äî Python, built on the Splunk SDK `splunklib.ai` agent primitives (`AgentState`, subagents, tool calls). Plans the run, routes to the right loop, invokes the self-heal engine.
- **Self-heal engine** ‚Äî generic loop: `diagnose(failure) ‚Üí select_patch() ‚Üí apply_patch() ‚Üí re-validate() ‚Üí repeat until clean or max_iters`. Artifact-agnostic, but intentionally **not** a free-form code editor. Patch application is done by deterministic patch functions from a constrained registry. The LLM produces diagnosis, rationale, and operator-facing explanation text only; it does not write raw file edits. This keeps the demo repeatable and the audit trail defensible.
- **MCP/SDK layer** ‚Äî MCP-app-token-authenticated client to the Splunk MCP server. Baseline tools: `splunk_run_query` for validation searches and `splunk_get_knowledge_objects` for existing knowledge-object discovery. Optional Splunk AI Assistant tools: `saia_generate_spl`, `saia_explain_spl`, and `saia_optimize_spl`. Direct `splunklib.client` is still used for ingestion/search setup where simpler than MCP.
- **Provenance ledger** ‚Äî append-only JSON log; one entry per patch: `{stage, iteration, failure, diagnosis, patch, rationale, validation_result, timestamp}`.
- **Splunk instance** ‚Äî `splunk/splunk:latest` in Docker, trial or Free license.

### 4.2 Auth note

Use the **encrypted token generated by the Splunk MCP Server app**. It is not a plain Splunk REST bearer token, and encrypted MCP tokens cannot be reused for direct Splunk API calls. OAuth for MCP exists, but it is Controlled Access / closed preview and is not a build dependency for this hackathon. Store the MCP endpoint and encrypted token in `.env`, never commit them.

`splunk_run_query` is intentionally bounded by the MCP server for safe, quick searches: about a 1-minute execution limit and about 1000 events returned. The onboarding fixture has 150 lines, so it is safely inside those limits.

### 4.3 Per-loop call flows

**Onboarding loop (Stage 1):**
1. Ingest raw log sample to Splunk via HEC (`splunklib` / HEC token).
2. Draft candidate extraction logic as inline SPL (`rex`, `eval`, `fieldformat`, aliases-as-eval) plus a candidate CIM data model mapping. Use `saia_generate_spl` only when Splunk AI Assistant is installed and the tool is available; otherwise use the local LLM/rule templates and still validate through MCP.
3. Run `splunk_run_query` through MCP against the ingested sample, validating the candidate extraction against real indexed events without writing `props.conf` / `transforms.conf` or reloading Splunk per iteration.
4. **Diagnose:** compare extracted fields vs. expected CIM fields. Count coverage (e.g. "4 of 9 fields extracted, `src` not mapped"). Use `splunk_get_knowledge_objects` where helpful to inspect existing field extractions, aliases, calculated fields, and data models.
5. **Patch:** deterministic onboarding patch functions adjust the inline `rex` / `eval` / alias candidate from a constrained set of known transformations. The LLM writes the diagnosis and rationale, not the patch bytes.
6. Re-run `splunk_run_query`. Repeat until CIM-clean. Flag any PII fields detected.
7. After convergence only, emit final `props.conf` / `transforms.conf` + provenance entries.

**AppInspect loop (Stage 2):**
1. Take a deliberately non-compliant app directory.
2. Run `splunk-appinspect inspect <dir> --mode test --data-format json --output-file result.json`.
3. **Diagnose:** parse `result.json`; extract each failure (check name, file, line, message).
4. **Patch:** deterministic AppInspect patch functions map known failure types to known repairs (remove forbidden `local/` config, remove `user-seed.conf`, disable/remove forbidden forwarding in `outputs.conf`, etc.). The LLM produces the human-readable diagnosis and rationale only; it does not edit files directly.
5. Re-run AppInspect. Repeat until zero failures or max_iters.
6. Emit clean app + provenance entries.

## 5. Seed fixtures (the demo's raw material)

Both loops need *deliberately broken* inputs so there is something to heal.

### 5.1 Onboarding fixture ‚Äî messy log sample

A custom-format log where field structure is non-obvious. **Recommended (on-brand, novel): a UPI / payment-gateway transaction log or GST e-invoice log.** Example shape:

```
2026-06-01T14:22:08+05:30 txn_id=UPI2026060114220812345 vpa=merchant@okhdfcbank amt=1499.00 status=SUCCESS rrn=412345678901 payer_vpa=customer@ybl gstin=24AAACC1206D1ZM
```

Designed so a naive extraction misses several fields and the CIM mapping (e.g. amount ‚Üí `amount`, status ‚Üí `action`/`status`) needs iteration. Include at least one field that should be flagged PII (e.g. `payer_vpa`).

Provide ~50‚Äì200 sample lines (well under the 500MB/day Free tier).

### 5.2 AppInspect fixture ‚Äî non-compliant app

A minimal Splunk app seeded with three deterministic AppInspect failures:

1. **Forbidden `local/` directory** ‚Äî config under `local/` triggers `check_that_local_does_not_exist`.
2. **Forbidden `user-seed.conf`** ‚Äî default login/password seeding triggers `check_user_seed_conf_deny_list`.
3. **Forbidden forwarding config** ‚Äî enabled `default/outputs.conf` triggers `check_if_outputs_conf_exists`.

Each has a known, scriptable fix, so the loop reliably reaches green on camera.

## 9. Risks & mitigations

- **Onboarding loop is the harder half** (regex gen + live-search validation has more moving parts) ‚Üí build AppInspect loop first as the guaranteed deliverable.
- **Docker Splunk first-boot is slow (~1‚Äì2 min to "healthy") and memory-hungry on Mac** ‚Üí pre-warm before any live demo; resource-limit the container.
- **Live demo gremlins** ‚Üí record a clean backup video; rules allow judges to score on video alone.
- **Over-promising the 5-stage platform** ‚Üí show two loops working for real; present stages 3‚Äì5 as labeled architecture. Judges reward honest scoping.
- **Direct overlap with Splunk's AI Intelligence Reports for AppInspect (early 2026)** ‚Üí name their feature explicitly in README/demo; position this as the layer above ("Splunk explains; we resolve, validate, remember"). The closed loop + provenance + onboarding integration is the defensible delta.
